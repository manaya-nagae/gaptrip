# GapTrip MVP 設計書

## 1. プロダクト概要

旅行の「妄想」(行きたい場所)と「現実」(行った場所)のギャップを記録・可視化するWebアプリ。Google Placesで場所を検索し、他のユーザーの口コミを参考にしながら「行きたい！」を登録。実際に訪れた後は満足度を記録し、期待とのギャップを楽しむ。日本地図で都道府県の制覇率も一目で確認できる。

## 2. ターゲットユーザー

- 旅行好きの個人
- 行きたい場所をリスト管理し、実際に行った後の振り返りを楽しみたい人
- 他の人のリアルな口コミを参考に新しい場所を発見したい人

## 3. 解決したい課題

- 行きたい場所が散在していて管理できない
- 旅行後に「期待と実際どうだったか」を振り返る手段がない
- 自分がどれだけ日本を巡ったか、一目で把握できない
- リアルな口コミを参考に行き先を決めたいが、既存サービスの口コミは信頼しにくい

## 4. 提供価値

| 価値 | 説明 |
|------|------|
| 発見 | Google Places検索と公開口コミで新しい場所を発見 |
| 可視化 | 日本地図で制覇率(XX/47)を一目で把握 |
| 記録 | 行きたい場所(WANT)と行った場所(WENT)を蓄積 |
| ギャップの楽しさ | 期待度 vs 満足度の差を「ギャップ」として振り返りを楽しくする |
| 共有 | 訪問記録を口コミとして公開し、他のユーザーの参考にしてもらえる |
| 導線 | 探す → Place詳細 → 行きたい登録 / 地図 → 都道府県 → スポット一覧 |

## 5. 技術スタック

- **フロントエンド**: Next.js 15 (App Router) / TypeScript / Tailwind CSS
- **データ取得**: Server Components + Prisma（DBへの直接アクセス）
- **更新処理**: Server Actions
- **ファイルアップロード**: Route Handler
- **ORM**: Prisma
- **DB**: PostgreSQL 16
- **認証**: NextAuth.js v5 (Credentials Provider / メール+パスワード)
- **場所検索**: Google Places API (New) — Autocomplete + Place Details
- **開発環境**: Docker Compose
- **デプロイ**: MVP期間はローカル開発のみ

- 口コミ公開/非公開（WENT記録のPublicReview化、取り消し可能）

## 8. 主要ユーザーフロー

### 8.1 新規登録 → 初回利用

```
ユーザー登録 → ログイン → マップ（全県グレー、制覇率0%）
→ ボトムナビで「探す」へ → 場所名検索 → Google候補から選択
→ Place詳細 → 「行きたい！」→ カテゴリ・期待度・メモ入力 → 保存 → 一覧に表示される
```

### 8.2 場所を発見してWANT登録する

```
探す画面 → 場所名を入力 → Google Places候補から選択 → Place詳細画面
→ 公開口コミを閲覧 → 「行きたい！」ボタン
→ カテゴリ・期待度★・メモを入力 → 保存
→ スポット一覧に戻る（WANTバッジ付きで表示）
```

### 8.3 現実を記録する（WANT → WENTフロー）

```
一覧画面 → スポットカードタップ → 詳細画面
→ 「行ってきた！」ボタン → 現実記録画面
→ 満足度★・感想・画像・訪問日を入力 + 「公開する」トグル → 保存
→ 詳細画面に戻る（WENTバッジ、ギャップスコア表示）
→ (公開ONなら) PublicReviewが自動作成され、Place詳細に口コミとして表示される
```

### 8.4 地図から探す

```
マップ画面 → 都道府県タップ → 一覧画面（その都道府県でフィルタ済み）
→ スポットカードタップ → 詳細画面
```

### 8.5 口コミを閲覧する

```
探す画面 → 場所名検索 → Place詳細画面 → 公開口コミ一覧を閲覧
```

### 8.6 口コミの公開/非公開を切り替える

```
スポット詳細画面（WENT済み）→ 「公開する」/「非公開にする」ボタン
→ PublicReviewの作成 or 削除
```

## 9. データモデル

### 9.1 User

| カラム | 型 | 必須 | 説明 |
|--------|------|------|------|
| id | String (cuid) | PK | |
| email | String | Yes, Unique | メールアドレス |
| password | String | Yes | ハッシュ済みパスワード |
| name | String | No | 表示名 |
| createdAt | DateTime | Yes (default: now) | 作成日時 |
| updatedAt | DateTime | Yes (auto) | 更新日時 |

### 9.2 Place

| カラム | 型 | 必須 | 説明 |
|--------|------|------|------|
| id | String (cuid) | PK | |
| googlePlaceId | String | Yes, Unique | Google Places place_id |
| name | String | Yes | 場所名（Google由来、最大500文字） |
| address | String | No | 住所（Google由来、最大500文字） |
| latitude | Float | No | 緯度 |
| longitude | Float | No | 経度 |
| prefectureCode | String | No | 都道府県コード（住所から導出） |
| googleTypes | String | No | Google Placesの分類（JSON文字列、参考情報） |
| photoReference | String | No | Google Places写真参照キー |
| createdAt | DateTime | Yes (default: now) | 作成日時 |
| updatedAt | DateTime | Yes (auto) | 更新日時 |

### 9.3 Spot

| カラム | 型 | 必須 | 説明 |
|--------|------|------|------|
| id | String (cuid) | PK | |
| userId | String | FK → User (onDelete: CASCADE) | 所有ユーザー |
| placeId | String | FK → Place (onDelete: RESTRICT) | 紐づく場所 |
| categoryId | String | FK → Category (onDelete: RESTRICT) | カテゴリ |
| status | Enum (WANT/WENT) | Yes (default: WANT) | ステータス |
| expectation | Int (1-5) | Yes | 期待度 |
| satisfaction | Int (1-5) | No | 満足度（WENT記録時に入力） |
| memo | String | No | メモ（WANT時の期待・気になること） |
| review | String | No | 感想（WENT記録時の振り返り） |
| imagePath | String | No | 画像ファイルパス（WENT記録時のみ設定） |
| visitedAt | DateTime | No | 訪問日（WENT記録時に入力、任意） |
| createdAt | DateTime | Yes (default: now) | 作成日時 |
| updatedAt | DateTime | Yes (auto) | 更新日時 |

**ユニーク制約**: `(userId, placeId)` — 1ユーザーが同一Placeに対して持てるSpotは1件のみ。

**ギャップスコア**: `satisfaction - expectation` で都度計算（-4 〜 +4）。カラムには持たない。

**Place参照**: スポット名・都道府県はSpotに持たず、関連するPlaceから取得する。

### 9.4 PublicReview

| カラム | 型 | 必須 | 説明 |
|--------|------|------|------|
| id | String (cuid) | PK | |
| userId | String | FK → User (onDelete: CASCADE) | 投稿者 |
| placeId | String | FK → Place (onDelete: RESTRICT) | 対象の場所（Spot.placeIdと同値。クエリ効率のため非正規化） |
| spotId | String | FK → Spot, Unique (onDelete: CASCADE) | 元のSpot（1 Spot → 0..1 Review） |
| displayName | String | No | 投稿者表示名（User.nameのsnapshot） |
| expectation | Int (1-5) | Yes | 期待度（snapshot） |
| satisfaction | Int (1-5) | Yes | 満足度（snapshot） |
| review | String | No | 感想（snapshot） |
| imagePath | String | No | 画像パス（snapshot） |
| visitedAt | DateTime | No | 訪問日（snapshot） |
| createdAt | DateTime | Yes (default: now) | 公開日時 |

**snapshotについて**: PublicReviewは作成時にSpotのデータをコピーして保持する。Spot側のデータが後から変更されてもPublicReviewには反映されない（公開時点のスナップショット）。再公開で最新化する場合は、一度非公開にしてから再度公開する。再公開時は新しいレコードが作成されるため `createdAt` は再公開時点の日時になる（意図的な仕様）。

**onDelete挙動**: `spotId` は `onDelete: CASCADE` のため、Spot削除時にPublicReviewも自動削除される。

### 9.5 Category（シードデータ）

| カラム | 型 | 必須 | 説明 |
|--------|------|------|------|
| id | String (cuid) | PK | |
| name | String | Yes | カテゴリ名 |
| sortOrder | Int | Yes | 表示順 |

初期データ:

| name | sortOrder |
|------|-----------|
| 食事 | 1 |
| 観光 | 2 |
| 宿泊 | 3 |
| アクティビティ | 4 |
| 買い物 | 5 |
| その他 | 6 |

### 9.6 Prefecture（コード定数）

テーブルではなく、アプリケーションコード内の定数として管理する。

```typescript
// 例
const PREFECTURES = [
  { code: "01", name: "北海道", region: "北海道" },
  { code: "02", name: "青森県", region: "東北" },
  // ...
  { code: "47", name: "沖縄県", region: "九州・沖縄" },
] as const;
```

## 10. 画面詳細仕様

### 10.1 ログイン画面 (`/login`)

**入力項目:**
- メールアドレス（テキスト、必須）
- パスワード（パスワード、必須）

**アクション:**
- ログインボタン → 認証成功でマップ画面へ、失敗でエラーメッセージ表示
- 「アカウントを作成」リンク → ユーザー登録画面へ
- 認証済みユーザーがアクセスした場合はマップ画面へリダイレクト

### 10.2 ユーザー登録画面 (`/register`)

**入力項目:**
- 名前（テキスト、任意）
- メールアドレス（テキスト、必須）
- パスワード（パスワード、必須、8文字以上）

**アクション:**
- 登録ボタン → アカウント作成後、自動ログインしてマップ画面へ
- 「ログインはこちら」リンク → ログイン画面へ
- 認証済みユーザーがアクセスした場合はマップ画面へリダイレクト

**バリデーション:**
- メール形式チェック
- メール重複チェック
- パスワード8文字以上

### 10.3 マップ画面 (`/`) — ホーム

**表示要素:**
- 日本地図（SVG）
  - 訪問済み都道府県（WENTスポットが1件以上）: テーマカラーで塗りつぶし
  - 未訪問: グレー
- 制覇率: 「XX / 47（XX%）」

**操作:**
- 都道府県タップ → スポット一覧へ遷移（`/spots?prefecture=XX`）

**初期表示:**
- ログインユーザーの全Spotデータを取得し、都道府県ごとにWENTの有無を判定
- 制覇率 = WENTスポットが存在する都道府県数 / 47（ログインユーザーのデータのみ対象）

### 10.4 探す画面 (`/search`)

**表示要素:**
- 検索バー（テキスト入力、Google Places Autocompleteに接続）
- 検索候補リスト（Google Placesの候補を表示）

**操作:**
- テキスト入力 → デバウンス（300ms）後にサーバーサイドプロキシ（`GET /api/places/autocomplete`）経由でGoogle Places Autocomplete APIを呼び出し → 候補リストを表示
- 候補タップ → Server Action `navigateToPlace(googlePlaceId)` を呼び出し → PlaceをDBにgetOrCreate → `/places/[id]` へリダイレクト

**初期表示:**
- 検索バーのみ表示。候補リストは空

> デバウンスはGoogle Places APIのリクエスト数抑制（コスト管理）とUXの両立のため。

### 10.5 Place詳細画面 (`/places/[id]`)

**表示要素:**
- 場所名（Place.name）
- 住所（Place.address）
- Google Placesの分類情報（参考表示）
- 写真（photoReferenceがあれば表示）
- 公開口コミ一覧（PublicReview）
  - 各口コミ: 投稿者表示名、期待度、満足度、ギャップスコア、感想、画像、訪問日
- 「行きたい！」ボタン（ログインユーザーがこのPlaceにSpotを未登録の場合）
- 「登録済み」表示（既にSpotがある場合、タップでSpot詳細へ遷移）

**「行きたい！」タップ時:**
- インラインまたはモーダルで入力フォームを表示:
  - カテゴリ（プルダウン、必須）
  - 期待度（★1〜5、必須）
  - メモ（テキストエリア、任意、最大1000文字）
- 保存 → Spot作成（status=WANT）→ スポット一覧へ遷移

**口コミ0件時:**
- 「まだ口コミがありません」

### 10.6 スポット一覧画面 (`/spots`)

**表示要素:**
- フィルタ
  - 都道府県プルダウン（初期値: すべて、マップからの遷移時は該当都道府県）
  - ステータスプルダウン（すべて / WANT / WENT）
- ソートプルダウン（更新日:新しい順 / 期待度:高い順 / 都道府県順）
- 検索件数（「全X件」）
- スポットカードリスト

**スポットカード表示項目:**
- サムネイル画像（画像がない場合はカテゴリに応じたプレースホルダー）
- スポット名（Place.name）
- 都道府県名（Place.prefectureCode由来）
- カテゴリ名
- ステータスバッジ（WANT / WENT）
- スコア: WANT時は「期待度★X」、WENT時は「期待度★X → 満足度★X（ギャップ+Y）」

**操作:**
- フィルタ・ソートのプルダウン変更 → URLクエリパラメータを更新（`/spots?prefecture=13&status=WANT&sort=expectation`）→ Server Componentが `searchParams` を読んで再レンダリング
- スポットカードタップ → スポット詳細へ

**0件時:**
- フィルタなしで0件: 「まだスポットがありません。「探す」タブから場所を見つけましょう！」
- フィルタありで0件: 「条件に一致するスポットが見つかりません」

### 10.7 スポット詳細画面 (`/spots/[id]`)

**表示要素:**
- 画像（あれば表示）
- 場所名（Place.name）
- 住所（Place.address）
- カテゴリ名
- ステータスバッジ（WANT / WENT）
- 期待度（★表示）
- メモ
- WENT時のみ追加表示:
  - 満足度（★表示）
  - ギャップスコア（+X / -X / ±0）
  - 感想
  - 訪問日
  - 公開状態（「公開中」/「非公開」）
- 登録日・更新日
- 「Place詳細を見る」リンク → `/places/[placeId]`

**アクション:**
- WANT時:
  - 「行ってきた！」ボタン → 現実記録画面へ
  - 編集ボタン → スポット編集画面へ
  - 削除ボタン → 確認ダイアログ → 削除 → 一覧へ
- WENT時:
  - 「公開する」/「非公開にする」ボタン → PublicReviewの作成 or 削除
  - 編集ボタン → スポット編集画面へ
  - 削除ボタン → 確認ダイアログ → 削除（PublicReviewも連動削除）→ 一覧へ

### 10.8 スポット編集画面 (`/spots/[id]/edit`)

登録済みSpotの情報を修正する画面。WANT・WENT問わず利用する。場所名・住所はPlace由来のため変更不可。

**表示要素（読み取り専用）:**
- 場所名（Place.name）
- 住所（Place.address）

**入力項目（WANT / WENT 共通）:**
- カテゴリ（プルダウン、必須、既存値プリセット）
- 期待度（★1〜5、必須、既存値プリセット）
- メモ（テキストエリア、任意、既存値プリセット）

**入力項目（WENT時のみ追加）:**
- 満足度（★1〜5、必須、既存値プリセット）
- 感想（テキストエリア、任意、既存値プリセット）
- 訪問日（日付選択、任意、既存値プリセット）

> 画像はこの画面では変更できない。現実記録画面（/record）でのWENT記録時のみ登録可能。

**アクション:**
- 保存ボタン → 更新 → スポット詳細へ戻る
- キャンセル / 戻る → スポット詳細へ

### 10.9 現実記録画面 (`/spots/[id]/record`)

WANTスポットをWENTに切り替える初回記録専用画面。WENT済みのSpotからはこの画面にアクセスできない（詳細画面にリダイレクト）。

**表示要素（参考表示）:**
- 場所名（Place.name、読み取り専用）
- 期待度（読み取り専用、★表示）

**入力項目:**
- 満足度（★1〜5、必須）
- 感想（テキストエリア、任意、最大1000文字）
- 画像（ファイル選択、任意、JPEG/PNG、最大5MB）
- 訪問日（日付選択、任意）
- 「公開する」トグル（任意、デフォルトOFF）

**アクション:**
- 保存ボタン → status を WENT に更新、満足度・感想・画像・訪問日を保存。公開ONならPublicReviewも作成 → スポット詳細へ
- キャンセル / 戻る → スポット詳細へ

### 10.10 プロフィール画面 (`/profile`)

**表示要素:**
- ユーザー名（未設定の場合は「未設定」）
- メールアドレス

**アクション:**
- ログアウトボタン → セッション破棄 → ログイン画面へ

## 11. 共通UI仕様

### 11.1 ヘッダー

`(main)` 配下の全画面に共通で表示:
- 左: GapTripロゴ（タップでマップ画面 `/` へ遷移）
- 右: ユーザーアバター（タップでプロフィール画面 `/profile` へ遷移）

ログイン・登録画面ではヘッダーは表示しない。詳細画面・編集画面・記録画面では戻るボタンを使用。

### 11.2 ボトムナビゲーション

4タブ構成:
- マップ（地図アイコン）→ `/`
- 探す（検索アイコン）→ `/search`
- 一覧（リストアイコン）→ `/spots`

## 12. 非機能要件

| 項目 | 内容 |
|------|------|
| 認証 | NextAuth.js Credentials Provider。セッション管理はJWT |
| パスワード保存 | bcrypt でハッシュ化 |
| 画像保存 | `public/uploads/` にローカル保存。ファイル名はUUID。Docker環境では `uploads` ディレクトリを bind mount で永続化し、コンテナ再作成後も画像が失われないようにする |
| 画像制限 | JPEG/PNG、最大5MB、1スポット1枚。WENT記録時（/record）のみ登録可能。登録後の画像変更はMVPに含まない |
| 画像削除 | Spot削除時は関連画像ファイルも合わせて削除する |
| Google Places API | APIキーはサーバーサイド専用（`GOOGLE_PLACES_API_KEY`）。Route Handlerでプロキシし、クライアントにキーを露出させない |
| データアクセス制御 | Spot: 一覧取得・制覇率集計はログインユーザーのSpotのみ対象。詳細・更新・削除はuserIdを条件に絞り込む。他ユーザーのSpot IDへのアクセス時は404を返す。Place / PublicReview: 全ユーザーが閲覧可能（認証は必要） |
| 論理削除 | 使用しない（物理削除） |
| 開発環境 | Docker Compose（app + PostgreSQL） |
| デプロイ | MVPではローカル開発のみ |

## 13. 制約事項

- 画像はローカルストレージ保存のため、デプロイ時にはストレージ戦略の見直しが必要
- Docker Compose運用では `public/uploads/` を bind mount で永続化する。Dockerfile に `RUN mkdir -p /app/public/uploads` が必要
- 画像はWENT記録時（/record）のみ登録可能。登録後の画像変更（差し替え・削除）はMVPに含まない
- パスワードリセット機能はMVPに含まない
- 1スポット1枚の画像制限（複数枚対応は将来）
- Google Places API (New) の利用が必須。APIキーの取得とBilling設定が前提
- 1ユーザーが同一Placeに対して持てるSpotは1件のみ（ユニーク制約）
- PublicReviewはスナップショット方式。Spotの編集内容はPublicReviewに自動反映されない
- Placeレコードは削除しない（Spot削除後も孤立Placeが残る）。孤立レコードのクリーンアップは将来対応

## 14. 未決事項

実装レベルの未決事項は全て解決済み。

| # | 項目 | 決定 | 理由 |
|---|------|------|------|
| 1 | Docker uploads ディレクトリの永続化方式 | **bind mount** | 個人ローカル開発環境でのファイル確認・バックアップの利便性を優先 |
| 2 | 他ユーザーのSpot IDにアクセスした場合の返却方針 | **404** | リソースの存在を他ユーザーに知らせない（情報漏洩防止） |
| 3 | 訪問日への未来日入力 | **許可する** | 「予定している旅行」の記録ユースケースが自然に存在するため |
| 4 | 同一PlaceへのSpot重複登録 | **禁止** | ユニーク制約(userId, placeId)で1ユーザー1Place1Spot。Place中心設計では場所がGoogle Placesで正規化されるため重複は不自然 |
| 5 | PublicReviewのスナップショット更新方式 | **手動（非公開→再公開）** | 自動同期は複雑。MVPではシンプルな手動更新で十分 |
| 6 | Place詳細のアクセス制御 | **認証必須だが全ユーザー閲覧可** | 口コミの発見価値を最大化。未認証はmiddlewareで弾かれる |
